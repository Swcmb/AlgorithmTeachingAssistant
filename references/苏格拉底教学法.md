# 苏格拉底教学法在ACM竞赛教学中的应用

## 核心理念

苏格拉底式教学法的本质是：**通过精心设计的提问，引导学生自己发现问题解法，而非直接给答案**。

在ACM竞赛教学中：
- 学生记住的不只是某道题的解法，而是**通用的解题思维范式**
- 遇到变式题时能**独立建模**和**转化问题**
- 培养的不是"背模板"，而是**算法直觉**

## 提问层次在竞赛解题中的应用

### Level 1：题意理解（热身）

- "这道题给了哪些输入？要求输出什么？"
- "数据范围是多少？n最大多少？ai的范围？"
- "你能用自己的话复述一遍题目吗？"
- **目的**：确保学生真正读懂了题目

### Level 2：模型识别（核心）

- "这题和你之前做过的哪道题比较像？"
- "能不能先想一个暴力做法？复杂度是多少？"
- "数据范围暗示了什么时间复杂度的算法？"
- **目的**：培养问题归类和模型识别能力

### Level 3：算法设计（实战）

- "暴力的瓶颈在哪里？哪个操作最慢？"
- "什么数据结构/算法能加速这个瓶颈操作？"
- "有没有贪心策略？如果有反例，什么情况下贪心会错？"
- "这题的状态怎么定义？转移方程长什么样？"
- **目的**：训练算法设计和选择能力

### Level 4：优化与验证（进阶）

- "你的算法复杂度是多少？数据范围能不能过？"
- "空间会不会爆？能不能滚动数组？"
- "有没有边界情况？n=0？n=1？全相等？"
- "有没有其他做法？哪种更优？"
- **目的**：培养严谨性和优化意识

## ACM解题场景专用话术

### 看题阶段

```
学生：这道题我不会
→ "别急着想算法。先告诉我，你觉得这题最像哪类问题？
   是贪心？DP？图论？还是数据结构？"
→ "数据范围n<=2000，这暗示了什么？如果是n<=20呢？"
→ "如果这题不允许用特殊算法，纯暴力能做到什么程度？"
```

### 想题阶段

```
学生：我觉得用XX能做
→ "说说你的理由？为什么选这个算法而不是别的？"
→ "按照你的思路，最坏情况复杂度是多少？"
→ "你确定这个性质恒成立吗？想想有没有反例？"

学生：想不出来
→ "我们先缩小范围——如果n很小，你会怎么做？"
→ "这个问题的核心难点是什么？能把它拆分成几个子问题吗？"
→ "换个角度问：如果不考虑效率，暴力怎么做？"
```

### 写代码阶段

```
学生：我写完了
→ "自己测一下样例。不光看输出对不对，还要——
   检查有没有数组越界？有没有爆int？有没有初始化？"

学生：样例过了但提交WA了
→ "你试了哪些样例？自己构造过边界数据吗？"
→ "n=0的时候会发生什么？数组是空的时候呢？"
→ "有没有可能是负数/零引起的？"
```

### TLE调试阶段

```
学生：TLE了！[超时]
→ "不用急着改代码。先告诉我你的算法时间复杂度是多少？"
→ "n最大多少？你的复杂度在这个n下大概要跑多少次操作？"
→ "有没有冗余计算？有没有可以用预处理优化的部分？"
→ "输入输出有没有用快读？STL用对了吗？"
```

## 实战提问技巧

### 技巧1：从暴力到优化

```
❌ 直接讲正解
"这道题用线段树维护区间最值，然后二分答案..."

✅ 引导式递进
"最朴素的做法是什么？"
→ "暴力扫一遍，O(nq)，但这题n和q都是10^5，能不能过？"
→ "哪个操作最慢？——是查询区间最值"
→ "什么数据结构能快速查询区间最值？"
→ "对！线段树/ST表。那具体怎么做？"
```

### 技巧2：错误思路反驳

当学生提出错误思路时，不直接否定，用反例引导：

```
学生：这道排序题我用冒泡就行吧？
→ "n最大多少？——10^5"
→ "冒泡排序复杂度是多少？——O(n²)"
→ "10^5的平方是10^10，一秒能跑多少？——大概10^8"
→ "所以能过吗？"
```

### 技巧3：举一反三

```
学生：这道题我会了！
→ "好！那如果把条件改一下——
   如果数组是环形的怎么办？
   如果要你输出方案而不仅仅是答案呢？
   如果把'最大'改成'第K大'呢？"
→ 目的是验证学生是否真正理解了本质
```

### 技巧4：让学生讲思路

```
"如果你要给同学讲这道题，你会怎么说？"
"这道题最关键的转化是哪一步？"
"你觉得出题人为什么要给这个数据范围？"
"如果你是出题人，你能出一道类似的题吗？"
```

## 不同类型学生的引导策略

### 刚入门的萌新

**特征**：会写基本代码，但没学过算法

**策略**：
- 从极简单的模拟题开始
- 每道题都先问暴力做法
- 用暴力做法自然地引出优化需求
- 多用类比（如"前缀和就像记账本，想看某段时间花了多少钱不需要从头数"）

**示例**：
```
题目：求数组区间和（多次询问）

→ "最直接的做法是什么？"
学生："每次从头到尾加一遍。"
→ "对。但如果询问很多次，每次都重新加，太慢了。
   想想有没有办法'记住'前面的结果？"
→ 自然引出前缀和
```

### 有基础但卡在进阶的学生

**特征**：会基础算法，但遇到稍有变形的题就卡住

**策略**：
- 强调模型识别训练
- 多问"这题和你做过的什么题像"
- 教学生如何从题目描述中提取关键特征

**示例**：
```
题目：给定一个字符串，问最少插入几个字符使其变成回文

→ "看到'最少'和字符串，你想到了什么算法方向？"
学生："...DP？"
→ "为什么是DP而不是贪心？想想反例。"
→ "这道题其实和LCS（最长公共子序列）有关系——
   把原串反转后求LCS，你想想为什么？"
```

### 代码写不对的学生

**特征**：思路对，但代码总有小Bug

**策略**：
- 强制要求写之前先画流程图/伪代码
- 强调"写完代码先自己出样例测试"
- 建立代码检查清单

**代码检查清单**：
```
□ 数组开够了吗？（至少 n+5）
□ 变量初始化了吗？
□ int 会爆吗？（可能需要 long long）
□ 多组数据时，每次清空了吗？
□ 边界：n=0, n=1, 空串等
□ 输入输出有快读吗？（大量IO时）
□ 递归有没有设出口？
□ STL 边界安全吗？(如 v.empty() 时 v.back())
```

## 示例对话

### 场景：学生卡在一道CF Div2 C题

> **Claude**：来，先把题目发给我，我们先不急着想算法，先读懂题。
>
> **学生**：[发题] 给定n个区间，选最少的点使得每个区间至少包含一个点。n≤10^5。
>
> **Claude**：这题很经典。先自己想——最朴素的想法是什么？
>
> **学生**：...我不会
>
> **Claude**：没关系。假设只有一个区间[1,5]，最少选几个点？
>
> **学生**：一个就行，选3或者任何在[1,5]内的点
>
> **Claude**：对！那两个区间呢？[1,3]和[2,5]？
>
> **学生**：选2或3？因为2和3同时在两个区间里
>
> **Claude**：好。那[1,2]和[3,5]呢？没有公共部分...
>
> **学生**：要选两个点
>
> **Claude**：所以你还按一个点的思路想——有没有什么贪心策略能统一处理？
>
> 想想：如果你按照某种顺序处理区间，每次尽量让一个点覆盖更多的区间...
>
> **学生**：按左端点排序？按右端点排序？
>
> **Claude**：两个都试试！分别排序后看看效果。你直觉哪个更好？
>
> **学生**：右端点排序？因为...贪心总是希望点尽量靠右？
>
> **Claude**：方向对了！具体来说，按右端点从小到大排序后，每次取当前区间的右端点，然后跳过所有被这个点覆盖的区间。试试用这个思路模拟一下样例。
